## [0.9.1] - 2026-07-23

A patch release, driven by a regression in 0.9.0. The element-memory bound that release introduced is sized for untrusted wire input, and buffa's own tooling applied it to the descriptor sets it reads — so codegen began rejecting schemas of a few hundred `.proto` files with `element memory limit exceeded`. Descriptor types are wide structs, giving a measured element footprint of roughly 6.3x the encoded size, so the 32 MiB default worked out to about 5 MB of descriptors. Tooling paths now decode under a 1 GiB bound. `DEFAULT_ELEMENT_MEMORY_LIMIT` is unchanged, so the protection for untrusted input is exactly as it was.

*Regenerate your code with a version-matched `buffa-codegen`.* The generated `descriptor_pool()` accessor now scales its decode bound to the descriptor bytes embedded in it; code generated by 0.9.0 keeps the old bound and the failure that comes with it. The `buffa-build` and `protoc` plugin fixes take effect on rebuild without regenerating.

API changes are additive only — `DescriptorPool::decode_with_options` and the new `buffa-codegen` decode helpers — so this is a drop-in upgrade from 0.9.0. MSRV is unchanged at 1.75.

### Changed

- **`UnknownFieldsView` holds its records behind a pointer, so a view carrying none costs a null** (#329).
  Every generated view embeds this type by value. Holding the records inline made each view an owning aggregate, which the compiler moves with out-of-line `memcpy` calls rather than inline vector stores, and that cost view-decode throughput even on payloads that never carry an unknown field. The state now lives behind a `Box`, allocated when the first unknown record arrives.
  Measured on `google_message1_proto3/decode_view` on quiesced bare metal, as the mean of four code-alignment layouts and reproduced on two CPU generations: a payload with no unknown fields decodes ~10% faster, and one carrying them at every nesting level ~2% slower, since each such view pays an allocation. A payload whose unknown fields all sit in one view falls between the two. Public API, wire behaviour and `Debug` output are unchanged.

### Fixed

- Field-scoped open-enum overrides now flow through `FieldDescriptor` into reflective binary and JSON decoding, so `DynamicMessage` preserves unknown enum values in singular, repeated, oneof, and map fields exactly like generated decoders. (#316)

- Descriptor decoding in buffa's own tooling no longer applies the untrusted-input element-memory bound, which rejected schemas of a few hundred `.proto` files with `element memory limit exceeded`. `buffa-build` and the `protoc` plugins now decode under a 1 GiB bound, overridable with the `element_memory_limit` plugin option or the `BUFFA_ELEMENT_MEMORY_LIMIT` environment variable; generated `descriptor_pool()` scales its bound to the descriptor bytes compiled into it. New `DescriptorPool::decode_with_options` lets any caller supply decode limits; `DescriptorPool::decode` stays bounded, since it may be handed an untrusted descriptor set. (#331)

- **Documentation: `DecodeOptions` bounds the binary codec only, and now says so** (#330).
  The 0.9.0 notes for `with_element_memory_limit` said "the owned, view and reflective (`DynamicMessage`) decoders are all bounded", which is true and was easy to read as codec-independent. It is not: JSON decoding runs `serde_json` straight into the generated `Deserialize` impls, which never receive a `DecodeOptions`, so a message parsed from JSON is bounded by none of the limits that bound the same message parsed from protobuf — and the element amplification is nearly as large there, `{}` being three JSON bytes for the footprint that costs two on the wire. The rustdoc on `DecodeOptions` and `with_element_memory_limit`, and the guide's decode-options section, now state the scope; the guide's table also lists `with_element_memory_limit`, which it had omitted. Bounding the JSON path is tracked in #330; no behaviour changes in this release.
